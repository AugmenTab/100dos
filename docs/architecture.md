# Architecture Decisions

Decisions recorded here were made during initial skeleton development. The goal is to prevent future confusion and preserve context that would otherwise exist only in chat history.

---

## System identity

**System ID:** `100dos`
**System title:** `100DOS`

The ID is lowercase and hyphenated to satisfy Foundry's module naming requirements. The title uses the canonical stylization of the game name.

---

## Foundry version target

The system targets the latest stable Foundry release at the time of implementation. There is no floor version pinning and no legacy compatibility shim. If a Foundry release introduces a breaking API change, the system updates rather than maintaining backwards compatibility.

---

## Build tool: Vite

Vite bundles TypeScript source into a single ES module (`100dos.mjs`) and copies static assets (templates, lang files, styles, the manifest) to `dist/`. The `scripts/build` and `scripts/watch` shell scripts invoke it inside the dev container.

Vite was chosen for its fast rebuild cycle and minimal configuration overhead relative to alternatives like Webpack or Rollup directly.

---

## TypeScript

All source code is TypeScript. The compiler is used for type checking only (`tsc --noEmit`) — Vite handles the actual transpilation. Foundry's own types are ambient; they do not ship as a package and must be declared locally as needed.

TypeScript was chosen to catch structural errors early, particularly around Foundry's document and sheet APIs, which are large and have subtle contracts.

---

## Development environment: Docker

The development Foundry instance runs in Docker via `docker-compose.yml`. Developers run `scripts/up` to start it and access Foundry in a browser at the configured port. npm and Node commands must be run inside the container, not on the host.

Docker was chosen to avoid requiring contributors to manage a local Foundry installation and to ensure a consistent runtime across machines.

---

## Actor types

The system defines three actor types: `pc`, `npc`, and `vehicle`. These are the only types expected for the foreseeable future. No catch-all or generic actor type exists — everything maps to one of these three.

---

## Setting content lives in separate modules

Game-world content (setting lore, gear lists, factions, etc.) does not live in this repository. It belongs in separate Foundry modules generated by Artificer. The system provides the rules engine and sheet structure; modules provide the content that runs on top of it.

This separation means the system can be used with any 100DOS setting without bundling content it may not need, and it allows content to be updated independently of the system.

---

## Artificer generates content modules

[Artificer](https://github.com/AugmenTab/artificer) is a web service that generates Foundry content modules for the 100DOS system. It packages compendium packs, internationalization files, and (eventually) custom stylesheets. It does not inject content at runtime — it produces static Foundry module packages that are installed and loaded normally.

Artificer defers to this system on all structural and naming decisions. If there is tension between what Artificer needs and what the system needs, the system takes precedence and Artificer adapts.

---

## Character creation wizard is post-MVP

No interactive character creation flow is planned during skeleton or initial sheet development. Characters are created by filling in sheet fields directly. A guided wizard may be added after the core sheet and data model are stable.
